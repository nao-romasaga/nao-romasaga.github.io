# レイドバトルシミュレーター 仕様書

`generate_damageRanking_v2.php` をベースにした、レイドバトル用ダメージランキング + パーティシミュレーター。
フロントエンド: `raid.html`（新規） / データ生成: `romasaga-tool-be/php/`（新規スクリプト）

---

## 1. レイドバトルの基本ルール

| 項目 | 仕様 |
|---|---|
| 敵の体力・精神 | 10固定 → 全hitがダメージカンストする |
| カンスト値 | 99,999,999 / hit |
| ターン数 | 3ターン固定 |
| BP | **無限**（サポーターからのBP供給前提。初期BP10だが管理不要） |
| 被弾 | 計算しない |
| OD | 使用可能（貯まった次ターンに発動） |
| スコア | パーティ5人の3ターン合計ダメージ |

> BP無限により現実には不可能なBP挙動も許容する。ただし実BP収支を計算し、不足する編成には**「要BP供給」マーク**を表示する。

## 2. ダメージ式

```
1hitのダメージ = 99,999,999 × (1 + Σ有効EF倍率) × Π(該当する剛体カットの残存率)
```

- EF分のダメージ（カンスト超過分）にも剛体カットは掛かる
- EFの有効判定は **hit単位** で突き合わせる（§4）

## 3. 剛体（敵アビリティ）

特定種別のダメージをカットする。**ボス設定としてユーザーが外部入力**する。

| 軸 | 判定フィールド | 値 |
|---|---|---|
| 直接 / 間接 | `Skill.csv` の `AttackMethod` | 1=直接(2709件) / 2=間接(3588件)。例外なし |
| 技 / 術 | `Skill.csv` の `SkillType` | 1=技 / 2=術 |

- UI: 直接/間接/技/術 の4種それぞれに**カット率を個別入力**（0% / 90% / 99% …）
- 複数該当時は**乗算**（直接90% + 技90% → 直接技は 0.1 × 0.1 = 1%）
- 現状ゲーム内は単一90%のみだが、複数持ち・99%にも対応できる設計とする

※ 調査済み: `AttackMethod=5` に見えた「練気ラッシュ+」はCSVクォート内カンマによるawkの列ズレで、正規パースでは全6297技が1か2。

## 4. エクストラフォース（EF）

### 4.1 倍率

| 強度 | 倍率 |
|---|---|
| 極小 | 0.12 |
| 小 | 0.25 |
| 中 | 0.5 |
| 大 | 0.75 |
| 特大 | 1.0 |

### 4.2 重複ルール

- **同名**（条件部分が同じ。例: Weak/小 と Weak/中）→ 効果が**上のもので上書き**
  - Weak/小付与スタイルとWeak/中付与スタイルが同居すると小は無駄になる（可視化対象）
- **異名** → **加算**。同時保有数は**無制限**
- 例: 40hitキャラに EF(Weak/中) → 40 × 1.5 = 60hit相当

### 4.3 有効条件（hit単位で判定）

| EF種別 | 有効になるhit |
|---|---|
| EF(Weak) | 弱点を突いたhit |
| EF(打) 等の属性 | 該当属性のhit |
| EF(直接) / EF(間接) | 該当AttackMethodのhit |
| EF(技) / EF(術) | 該当SkillTypeのhit |
| EF(OD) | OD行動のhit |
| EF(女性) 等のキャラ条件 | 性別/シリーズ/キャラ名/特定技が一致するスタイルのhit（`$EF_CONDITION_MAP` 参照） |
| EF(斬・陽) 等の複合 | AND条件で評価（既存実装踏襲） |

### 4.4 付与タイミング（ターン内の細則）

| 付与タイミング | 有効になる時点 |
|---|---|
| バトル開始時 / ターン開始時 | 全員の行動前に発動 |
| 「攻撃前、〜に付与」と明記 | その攻撃のhitから乗る |
| 攻撃時 | **同じ攻撃の以降のhitから**（hit単位で累積） |
| 攻撃後 | 次の行動から |
| ターン終了時 | 次のターンから |

効果ターン（1ターン/2ターン/バトル中永続）はJS側のタイムラインで正確に管理する。

### 4.5 付与経路（PHP側で抽出が必要）

1. アビリティによる直接付与（バトル開始時 / ターン開始時 等）
2. 補助技による付与（limit 1回・リキャスト・**先頭キャラにしか使えない**制約あり）
3. 攻撃技に付随する付与（全体対象のものはサポート効果として他者に乗る）
4. **チェーン付与**: 味方にアビリティ付与 → そのアビからEF付与
5. **敵経由**: 敵にアビリティ付与 → 特定条件で味方全体にEF付与 / 敵に付与したアビから「敵全員(=プレイヤー側)」にEF付与
6. チェーンは**2段ネストあり** → 再帰抽出で対応
7. **武器による付与**: 特定キャラが装備するとEFが付く武器がある
   - 例: `711102793 邪神憑りし者の大刀` — キャラクターが飯綱こよみの場合、自身に「エクストラフォース(邪神憑りし者/中)」を付与
   - 既存実装あり: 武器マスタの `ExtraForce` フィールド + `EF_CONDITION_MAP` / `_isEfNamedConditionMet()`（`damageRankingFunction_v2.inc`）
   - レイド版では**武器EFマスタ**として抽出し、JS側でアタッカーの武器種×キャラ条件が一致すれば自動装備扱いで適用（UIでON/OFF切替可能にする）
   - カンスト環境ではEFがそのまま倍率になるため影響大

付与対象のバリエーション: 自身 / 味方単体（**先頭=スロット1に付与**） / 味方生存者全体 / 同一武器種 / 性別条件 / 特定技条件。

## 5. 行動モデル（オートAI前提）

レイドは高速オート周回が主目的のため、**オートAIの挙動**をシミュレートする。

### 5.1 技選択の優先度

```
消費BP最大の技 > 継承技 > 技3 > 技2 > 技1
```

- リキャスト中・limit切れの技はスキップして次候補
- 補助技と攻撃技が両方使える場合も同じ優先度に従う
- **技封印（「使用しない」設定）**: 技ごとにOFF可能。封印はオートAIの選択を制御する手段
  - 例: ゆめはは技2を封印することで補助技を連打できる
  - アタッカー側でも封印で行動が変わるため**封印設定の組み合わせを列挙**する

### 5.2 行動パターンの決定性

BP無限のため、行動パターンは以下で**決定的**に決まる:

```
パターン = f(封印設定の組み合わせ × 継承技の選択 × ODタイミング × 敵弱点)
```

既存のBP管理ベースのパターン列挙は不要 → 生成処理は大幅に軽量化。

### 5.3 OD（オーバードライブ）

- ゲージ蓄積: **1攻撃 +16%**（暫定）+ ODゲージ回復アビ + サポーターの満タン付与
- ゲージ100%になった**次ターン**に発動。使用後ゲージは0から再蓄積（複数回発動あり得る）
- OD時の行動: 行動回数は増えない。技選択は通常の優先度に従う
- OD時の効果: EF(OD)が乗る / OD攻撃時トリガー発動（**今回はスルー可**）/ 全ステ+20%・ダメージアビ+50%（カンスト固定なので無視）
- OD使用者は**そのターンの先頭で行動**（その時点で乗っているEFのみ有効）

### 5.4 行動順（ご都合主義の簡略ルール）

```
OD使用者（ファスト→通常→ディレイ） → ファスト → 通常 → ディレイ
```

- ランキング/シミュレーターでは **スロット1（先頭）は常に最後に行動**（補助の後に動く。OD時を除く）
- 同区分内は素早さ降順（最大ステータス: 裏・昇段込み。バフによる素早さ変動は無視）

### 5.5 確率発動・hit数変動

- 確率発動の追撃・EF付与等は**発動前提**で計算
- hit数変動（`MinActionTime ≠ MaxActionTime`）は**平均値と最大値の両方**を計算し、UIで切替可能にする

### 5.6 サポーターの挙動

- 補助技を使わないターンは**通常攻撃**する（攻撃時EF付与・行動回数カウントに影響）
- フィラー継承（攻撃抑制用の適当な補助技継承。例: A.Jにチューンナップ）は
  **「補助技継承」という抽象アクション**として扱う（実名は不問）
  - ただし**継承可能な補助技がないキャラは不可**なので、キャラごとに有無をチェック
- **攻撃するスタイルのhitは全てスコアに入る**（純サポーターは補助技連打により自然にhit数ゼロに近づく。
  ハイブリッド型は自身の攻撃hitがスコアに入りつつ、攻撃パターンに付随するEF付与イベントが他者にも乗る）

## 6. 敵弱点と行動パターン変種

### 6.1 弱点が行動に影響するケース

Weak反応（Weak時BP回復 / Weak追撃 等）を持つスタイルのみ、弱点によって行動が変わる。

### 6.2 生成方式: 攻撃属性の部分集合

- **Weak反応スタイル**: そのスタイルの攻撃属性集合の**全部分集合**ごとにパターン生成
  （例: {斬,陰,雷} → 2³ = 8変種。陰弱点/雷弱点/陰雷弱点いずれも正確にマッピング可能）
- **非Weak反応スタイル**: 行動は弱点に依存しないため**1変種のみ**
- JS側: `ボス弱点集合 ∩ スタイル攻撃属性集合` → 該当変種を選択
- **EF(Weak)の有効判定はJS側**で hit属性 × ボス弱点集合 により行う（複数弱点に近似なしで対応）

## 7. 生成対象スタイルの範囲

| 対象 | 範囲 |
|---|---|
| アタッカーパターン | **2025年10月以降リリース**のスタイルのみ（処理軽減）。かつ3ターンで**30hit未満は除外** |
| サポートプロファイル | **全スタイル**。一度計算して古いスタイルにEF付与が存在しないことが分かったら、再計算時に除外レンジを決める |

リリース日は `data_StyleGacha.csv` の並び順から判定（要確認: 日付フィールドの有無）。

## 8. アーキテクチャ

### 8.1 役割分担

```
【PHP・事前生成】
  - スタイル × 弱点部分集合 × ODパターン変種（なし/T1/T2/T3/組合せ）の
    3ターン行動パターン + hit明細 JSON
  - 全スタイルのサポートプロファイル
  - スタイルメタデータ

【JS・動的計算（raid.html）】
  - ボス設定（弱点・剛体）の適用
  - パーティ5人のEFタイムライン合成（同名上書き → hitごとに有効EF判定）
  - ODゲージ収支検証 → 実現可能なODパターン変種の採用
  - 補助技プランの全探索（3ターン × 技数なので小規模）
  - スコア集計・ランキング・増分計算
```

EF・剛体・弱点・パーティの組み合わせ爆発を全てJS側に逃がすことで、ランキング再生成不要・画面即時切替を実現する。

### 8.2 hit明細レコード（PHP出力）

```
{ ターン, 技ID, hit数(平均/最大), 属性(複合可), 直接間接, 技術, OD有無,
  自己EF付与イベント(種類/倍率/タイミング/効果ターン/対象/条件) }
```

### 8.3 サポートプロファイル（PHP出力）

```
{ スタイルID,
  EF付与イベント: [{ EF名(条件部), 強度/倍率, 付与タイミング, 対象, 対象条件,
                     効果ターン, 経路(アビ/補助技/攻撃技付随/チェーン),
                     BPコスト, limit, リキャスト }],
  ODゲージ供給: [{ タイミング, 量(満タン/+n%) , 対象 }],
  BP供給有無,
  継承可能補助技の有無（フィラー用） }
```

### 8.3b 武器EFマスタ（PHP出力）

```
{ 武器ID, 武器名, 武器種, EF名(条件部), 強度/倍率, 装備条件(キャラ名 等) }
```

### 8.4 スタイルメタデータ（PHP出力）

```
{ スタイルID, キャラ名, 武器種, 性別, シリーズ, 素早さ(最大ステ), 技ID一覧, リリース日 }
```

EFキャラ条件（同一武器種/性別/特定技 等）の判定に使用。

### 8.5 配信形式

高速に処理できればよい。候補: **サマリ1ファイル（ランキング初期表示用）+ hit明細は個別/分割ファイルの遅延ロード**、gz配信。実装時にサイズを見て決定。

## 9. UI（raid.html）

1ページ構成。既存ページと同じスタック（jQuery + 素のJS + 既存CSS）。

### 9.1 上部: ボス設定フィルタ

- 弱点属性: **複数選択可**
- 剛体: 直接/間接/技/術 × カット率の個別入力

### 9.2 中央: アタッカーランキング

- サポートなし時のスタイル単体スコア順
- 表示項目:
  - **ターンごと**の行動・使用技・**技ごとの属性hit数**
  - 乗っているEFの内訳（種類・倍率・どのhitに有効か）
  - スコア（ダメージ）
  - **総行動回数**（多い=周回が遅い。補助技継承で追撃が消える効果も可視化）
  - 要BP供給マーク（実BP収支が不足する場合）
  - 平均hit / 最大hit の切替

### 9.3 下部: パーティシミュレーター

- 5スロット。各スロットに**スタイルを選択**（アタッカー/サポーターの定義なし）
- **並び替え可能**（スロット1=先頭: 単体EF・補助技の対象、最後に行動）
- 選択UI: アタッカーランキングから選択 / サポーターランキングから選択 / キャラ→スタイルの自由選択 の3系統
- **サポーターランキング** = 現在のパーティ構成に対して空きスロットに入れた時の**スコア増分**順
  （パーティが空の状態では「スロット1の最強アタッカー単体に対する増分」）
  - 候補は**EF付与可能なスタイルのみに事前フィルタ**して計算する（サポートプロファイル生成時に
    EF付与イベントを持つスタイルを抽出 → 増分計算の母集団を絞り性能問題を回避）
- サポーターを入れるとランキング/スコアが即時再計算される
  （例: 全体EF(打/大)付与スタイルを入れる → 打属性hitのスコアが1.75倍に）
- 同名EFの無駄（上書きされる側）の可視化

### 10.0 フェーズ2の実装メモ（generate_raidPattern.php）

- 行動エンジンは `filterUseSkillPattern` を使わず新実装。追撃展開は既存の
  `addTsuigekiFromSkill` / `getThisTurnTsuigeki` を再利用
- **戦闘中アビリティ付与（鬼八のWeak追撃等）への対応**: `setTimingAttrs` + `addAbilityFromAbilityWithBuff` を
  バトル開始時/ターン開始時/攻撃時/ターン終了時に発火させ、`TSUIGEKI_ABILITY` への動的登録を再現
- **確率発動=発動前提**: `$PER_TABLE` を全true化 + `$MAX_TURN=5` 扱い
- **出力のdedupe戦略**: turnsにODフラグを持たせない（JS側がconfigsのodTurnsから該当ターンのhitにODマークを付与）。
  同一行動列のconfigs（封印/継承/OD/弱点の組み合わせ）を1変種に集約 → 43MB→1.2MBに圧縮
- 攻撃トリガーの発火粒度は「行動単位」（hit単位ではない）。「〜のたび」系の回数制限付きアビ付与は
  hit単位より少なめにカウントされる近似がある（v1割り切り、検証フェーズで問題が出たら精緻化）
- 弱点部分集合は常に全列挙し、非Weak反応スタイルはdedupeで自然に1本へ収束させる方式に変更
  （TSUIGEKIキー検査では「アビ付与経由のWeak追撃」を見逃すため）
- **要BP供給（bpNeed）**: 各configに `b` として記録。BPモデル（ヴィクトール実機検証で確定）:
  - 初期BP10 + **自然回復+3/ターン（2ターン目以降のターン開始時）**
  - 自前BP回復: `healBp()` のバトル開始時/ターン開始時/ターン終了時 + **攻撃時・攻撃命中時は行動（追撃含む）ごとに発火** + 技ModのBP回復平均値
  - **`$LONG_BATTLE_ABILITY_ATTR`（長期戦用のBP+4/ターン等のパーティ仮定）はレイドでは無効化**（混入するとbpNeedが過小評価される）
  - 確率付きBP回復は保守的に不算入。0=自給自足可
  - 検証: ヴィクトール = T1:15-15、T2:13-14(不足1)、T3:13-14(不足1) → bpNeed=2 ✅
  - config採用優先度は「要BP供給が少ない > 継承なし > 封印数が少ない」
- 実機検証済み: アーニャ2024-12版（新春乱れ咲き3連+技攻撃時追撃×2=33hit、要BP供給16）、
  アーニャ2026-06版（33hit/15行動）、鬼八4スタイル（OD追撃分岐含む）が実機挙動と一致
- **付与アビ内の追撃はトリガー発火時に `addAbilityFromAbilityWithSkillPattern` も呼ぶ必要がある**
  （`addAbilityFromAbilityWithBuff` はダメージ系のみで main=追撃 を処理しない）。イシス「音の力」で発覚
- **効果ターンの減衰**: ターン終了時に `decrementBuff()` + `decrementTsuigeki()` を呼ぶ
  （鬼八のチェインアーツ＝付与1ターンがT1で切れる）
- **リキャストはオフバイワン注意**: リキャストn = nターン後に再使用可（T1使用・リキャスト2→T3使用可）。
  残カウントはターン開始時デクリメント方式なので n をそのままセットする
- 実機一致の追加確認: イシス 13/12/13=38hit（T3にメロディ再使用で追撃付け直し）、鬼八2026-01版 40/32/32=104hit
- **状態条件つきEFの静的評価**（サラ[厄]の厄の目覚めで発覚）:
  レイドは被弾なし=状態が戦闘中に変化しないため、抽出時にdamageRanking共通の `_isSingleConditionMet` で
  状態条件（「闇に染まる回顧」状態でない場合 等）を静的評価し、恒久不成立イベントに `inactive=1` を付与。
  JS側は inactive をスキップするだけで条件評価ロジックの二重実装を回避（全1990スタイルで39イベントが不成立判定）
- 既知の未接続: `使用回数制限回復` はレイドエンジン独自のlimit管理（state.useCount）に未接続。
  状態条件つき回復（厄の目覚め等）が必要になったら raidSelectSkill 側に配線する

## 11. 決定事項の経緯メモ

- スコア=ダメージ。カンスト固定なので実質 hit数 × EF倍率 × 剛体 の勝負
- 重要なのは攻撃回数・hit数・EFの可視化（ダメージ計算の精密さは不要）
- BP無限の採用理由: サポーターのBP供給前提 + 生成処理の軽量化
- ご都合主義の行動順を採用（先頭は最後に行動）: ランキングとして最大値を出すため
- 部分集合方式の採用理由: 複数弱点ボスに近似なしで対応するため
